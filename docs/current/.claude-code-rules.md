### All Documentation Must Be Versioned

Every markdown file in this project MUST include a header with:

```markdown
**Version**: X.Y.Z
**Last Updated**: [Month Day, Year] at [HH:MM] UTC
**Status**: [Draft | Active | Deprecated | Archived]
```

### When Creating New Documentation

1. Always include version header (start at 1.0.0)
2. Always include last updated timestamp with date AND time
3. Include status field
4. Place in appropriate location:
   - Root: ONLY `README.md` and `CLAUDE.md`
   - `docs/current/`: All other active documentation
   - `docs/archive/`: Superseded documents

### When Modifying Documentation

1. Update the "Last Updated" timestamp
2. Increment version number:
   - Patch (0.0.X): Typo fixes, clarifications
   - Minor (0.X.0): New sections, significant updates
   - Major (X.0.0): Complete rewrites, major restructuring
3. If the document is being superseded, move to `docs/archive/` and update status to "Archived"

### Documentation Hierarchy

| Document | Location | Purpose |
|----------|----------|---------|
| `README.md` | Root | Master document, project overview |
| `CLAUDE.md` | Root | Claude Code auto-loaded instructions (read every session) |

### Preventing Outdated Documentation Issues

1. **Never reference archived documents** for implementation decisions
2. **Always check document dates** before using specifications
3. **If a document is >30 days old**, verify it still reflects current implementation
4. **Archive documents immediately** when they become outdated
5. **Cross-reference `docs/current/MEMORY.md`** as the source of truth for current state

## Rule 2: Auto-Commit

### Commit at End of Each Task

After completing any task or making changes:

1. Stage all changes: `git add .`
2. Create a descriptive commit with proper message format
3. Push to GitHub: `git push origin main`

### Commit Message Format

```
Brief description of changes

- Bullet point list of specific changes
- Include technical details when relevant
- Reference any issues or features

ðŸ¤– Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude Opus 4.5 <noreply@anthropic.com>
```

### When to Skip Commits

- If no meaningful changes were made
- If only reading/investigating code
- If user explicitly requests no commit

## Rule 3: Git Safety

- NEVER force push to main
- NEVER amend commits from other developers
- Always use descriptive commit messages
- Follow conventional commit format when appropriate
- Verify remote status before pushing

## Rule 4: Code Quality

### Before Completing Any Task

1. Run `npx vue-tsc --noEmit` to verify TypeScript compiles
2. Ensure no regressions in existing functionality
3. Use existing patterns and composables when possible
4. Follow the established component structure

### Type Safety

- All new code must be TypeScript
- Use types from `types/index.ts` and `types/enhancements.ts`
- Avoid `any` type unless absolutely necessary

### Mock Data Files

All data files in `public/data/` must:

## Rule 6: Styling Standards (CRITICAL)

### MANDATORY: Use Rialtic Theme Colors

This codebase is aligned with the console-ui design system. **ALL new code MUST use the Rialtic color palette.**

Reference: `docs/current/STYLE_GUIDE.md`

### Prohibited Color Classes

**NEVER use these Tailwind default colors:**

| Prohibited | Required Replacement |
|------------|---------------------|
| `gray-*` | `neutral-*` |
| `green-*` | `success-*` |
| `red-*` | `error-*` / `error-light` / `error-dark` |
| `blue-*` | `secondary-*` (accents) or `primary-*` (CTAs) |
| `indigo-*` | `primary-*` |
| `yellow-*` | `warning-*` |

### Required Color Usage

```
Primary (Rialtic Blue #3F52BC):
- primary-600: Primary buttons, active nav items
- primary-700: Hover states on primary buttons
- primary-50/100: Light backgrounds for primary sections

Secondary (Cyan #28B7F0):
- secondary-600: Links, secondary actions
- secondary-50/100: Info backgrounds, accents

Neutral (Grays):
- neutral-900: Headings, primary text
- neutral-700: Body text
- neutral-500: Secondary text
- neutral-200: Borders
- neutral-50: Backgrounds

Success (Green #31AA8F):
- success-600/700: Success text
- success-50/100: Success backgrounds

Error (Red #E45D7B):
- error-600/700: Error text
- error-light: Error backgrounds

Warning (Yellow #F6BD60):
- warning-600/700: Warning text
- warning-50/100: Warning backgrounds
```

### Typography Shortcuts

Use semantic shortcuts instead of raw Tailwind classes:

| Shortcut | Usage |
|----------|-------|
| `h1`, `h2`, `h3` | Headings |
| `body-1`, `body-2` | Body text |
| `subtitle-1`, `subtitle-2` | Subtitles |
| `text-large-1` to `text-large-5` | Display text |

### Component Shortcuts

Use these shortcuts for consistency:

| Shortcut | Usage |
|----------|-------|
| `btn btn-filled` | Primary button |
| `btn btn-outlined` | Secondary button |
| `btn-text` | Text button |
| `form-input` | Form inputs |
| `card` | Card containers |
| `checkbox`, `radio` | Form controls |

### Icon Standards

Use Heroicons as the primary icon set:
```html
<Icon name="heroicons:icon-name" class="w-5 h-5" />
```

### Before Submitting Code

1. **Search for prohibited colors** - Ensure no `gray-`, `green-`, `red-`, `blue-`, `indigo-`, `yellow-` classes
2. **Use semantic colors** - All colors should convey meaning (primary, secondary, success, error, warning, neutral)
3. **Check STYLE_GUIDE.md** - Reference for correct patterns

### Example: Correct vs Incorrect

```html
<!-- INCORRECT - DO NOT USE -->
<div class="bg-gray-100 text-gray-900 border-blue-500">
  <span class="text-green-600">Success</span>
  <span class="text-red-600">Error</span>
</div>

<!-- CORRECT - USE THIS -->
<div class="bg-neutral-100 text-neutral-900 border-secondary-500">
  <span class="text-success-600">Success</span>
  <span class="text-error-600">Error</span>
</div>
```
